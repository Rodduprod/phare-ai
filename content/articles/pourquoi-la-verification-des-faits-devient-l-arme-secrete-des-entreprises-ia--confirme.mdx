# Vérification des faits en IA : comment ne pas se faire avoir comme un bleubite

On ne va pas se mentir : l’IA ment. Pas par malveillance, non, juste parce qu’elle a été entraînée sur des données qui valent ce qu’elles valent. Et aujourd’hui, la vérification des faits n’est plus un luxe, c’est un avantage compétitif. Les entreprises qui savent distinguer le vrai du faux dans leurs pipelines IA sont celles qui survivent. Les autres ? Elles finissent en Une du *Journal du Net* avec un titre du genre *"Notre chatbot a conseillé à un client de boire de l’eau de Javel"*.

Alors comment ça marche, techniquement ? Spoiler : ce n’est pas juste une couche de RAG en plus.

---

## Les fondements techniques : pourquoi votre LLM a besoin d’un garde-fou

Un modèle de langage, aussi gros soit-il, reste un autocompléteur sophistiqué. Il ne *comprend* pas, il prédit. Et quand on lui demande de vérifier une info, il fait ce qu’il sait faire de mieux : inventer une réponse plausible.

**Le problème de base** : les LLMs n’ont pas accès à des sources en temps réel, et leur connaissance s’arrête à leur date d’entraînement. Même avec du fine-tuning, un modèle ne peut pas savoir si une info est vraie ou fausse, il peut juste estimer sa probabilité.

La solution ? **Coupler le modèle avec des systèmes externes de vérification.**

### Architectures clés

1. **RAG + Fact-Checking**
   Le Retrieval-Augmented Generation (RAG) permet d’enrichir les réponses du LLM avec des données externes. Mais attention : si la source elle-même est fausse, le RAG ne fait que propager l’erreur.
   **Solution** : intégrer un module de vérification en amont.
   Exemple d’architecture :
   - Un **classifieur de crédibilité** (entraîné sur des datasets comme [FEVER](https://fever.ai/) ou [Vitamins](https://github.com/google-research-datasets/vitamins)) évalue la fiabilité des sources avant qu’elles n’atteignent le LLM.
   - Un **système de cross-checking** compare les infos avec plusieurs sources indépendantes.

   ```python
   # Exemple simplifié de pipeline avec vérification
   def verify_fact(claim, sources):
       credibility_scores = [classifier.predict(source) for source in sources]
       if min(credibility_scores) < THRESHOLD:
           return "Source non fiable, vérification nécessaire"
       else:
           return cross_check(claim, sources)

   def cross_check(claim, sources):
       # Implémentation basique : vérifier la cohérence entre sources
       return all(claim in source for source in sources)
   ```

   *Oui, c’est simpliste. Non, ça ne suffit pas pour la prod.*

2. **Modèles spécialisés dans le fact-checking**
   Des modèles comme **FactCC** ou **MacBERTh** (pour le chinois) sont entraînés spécifiquement pour détecter les fausses infos.
   **Problème** : ils sont souvent limités à un domaine (politique, santé, etc.) et nécessitent un fine-tuning constant.

3. **Graphes de connaissances + Inférence**
   Utiliser des **knowledge graphs** (comme Wikidata) pour valider les affirmations.
   Exemple : si le LLM affirme *"Paris est la capitale de la France"*, on vérifie dans le graphe que :
   - Paris est bien une ville
   - La France est bien un pays
   - Paris est bien la capitale de la France
   **Limite** : les graphes de connaissances sont statiques. Si une info change (ex : un pays change de capitale), il faut mettre à jour le graphe.

4. **Systèmes hybrides : LLM + Règles métiers**
   Pour les entreprises, la vérification ne se limite pas aux faits publics. Il faut aussi vérifier contre :
   - Les **données internes** (ex : base de clients, inventaires)
   - Les **règles métiers** (ex : "un client ne peut pas avoir un solde négatif")
   **Exemple concret** : [Comment Airbus protège ses plans d'avion avec de l'IA (sans tout casser)](https://lelabo.ai/articles/comment-airbus-utilise-l-ia-pour-proteger-ses-plans-d-avion-comme-une-grand-mere-son-livre-de-recettes--confirme) montre comment intégrer des vérifications métiers dans un pipeline IA.

## Implémentation : comment passer de la théorie à la prod

### Étape 1 : Choisir ses sources
Pas de vérification sans données fiables. Les options :
- **APIs de fact-checking** : [ClaimReview](https://schema.org/ClaimReview) (Google), [FactCheck.org](https://www.factcheck.org/)
- **Bases de données structurées** : Wikidata, DBpedia
- **Sources internes** : CRM, ERP, bases de connaissances métiers

**Piège à éviter** : se reposer sur une seule source. Même Wikipedia peut se tromper.

### Étape 2 : Construire le pipeline
Un exemple d’architecture robuste :

1. **Prétraitement** : Nettoyage de la claim (suppression des stop words, normalisation)
2. **Récupération** : RAG pour trouver des sources pertinentes
3. **Vérification** :
   - Vérification syntaxique (ex : "La Terre est plate" → incohérence avec les données géographiques)
   - Vérification sémantique (ex : "Elon Musk est PDG de Tesla" → cohérent avec les données récentes)
   - Vérification temporelle (ex : "La France a gagné la Coupe du Monde en 2022" → faux, c’était l’Argentine)
4. **Agrégation** : Combinaison des scores de vérification
5. **Post-traitement** : Génération d’une réponse avec niveau de confiance

```python
# Exemple de pipeline avec LangChain (simplifié)
from langchain.chains import RetrievalQA
from langchain.vectorstores import FAISS
from langchain.llms import OpenAI

# 1. Chargement des sources fiables
vectorstore = FAISS.load_local("wikipedia_fiable", embeddings)

# 2. Chaîne de RAG
qa = RetrievalQA.from_chain_type(
    llm=OpenAI(),
    chain_type="stuff",
    retriever=vectorstore.as_retriever()
)

# 3. Vérification avec un classifieur externe
def verify_claim(claim):
    # Appel à une API de fact-checking
    response = requests.get(f"https://api.factcheck.org/verify?claim={claim}")
    return response.json()["is_true"]

# 4. Intégration dans le pipeline
def safe_qa(question):
    answer = qa.run(question)
    if not verify_claim(answer):
        return "Désolé, je ne peux pas confirmer cette information."
    return answer
```

**Problème** : la latence. Chaque étape ajoute du temps. Pour un chatbot en temps réel, il faut optimiser.

### Étape 3 : Optimisations
- **Cache** : Stocker les résultats de vérification pour les claims fréquentes
- **Parallélisation** : Exécuter la récupération et la vérification en parallèle
- **Modèles légers** : Utiliser des modèles de vérification distillés (ex : DistilBERT pour le classifieur)

## Limitations : pourquoi ça ne marche pas à tous les coups

1. **Le problème des "faits" subjectifs**
   *"Ce film est le meilleur de l’année"* → Impossible à vérifier objectivement.
   **Solution** : Se limiter aux faits vérifiables (dates, chiffres, noms).

2. **Les données en temps réel**
   Un LLM entraîné en 2023 ne sait pas qui a gagné la Coupe du Monde 2026.
   **Solution** : Intégrer des APIs d’actualités (ex : NewsAPI, Twitter API).

3. **Les biais des sources**
   Si toutes vos sources viennent de *Breitbart*, votre système de vérification aura un biais.
   **Solution** : Diversifier les sources et pondérer par crédibilité.

4. **Le coût**
   Vérifier chaque claim coûte cher en compute et en API calls.
   **Solution** : Prioriser les claims critiques (ex : infos financières, médicales).

5. **Les "unknown unknowns"**
   Comment vérifier une info que personne ne connaît encore ?
   **Solution** : Marquer comme "non vérifié" et alerter un humain.

## Recherche & évolutions futures

### Ce qui arrive bientôt
1. **Modèles de vérification multilingues**
   Aujourd’hui, la plupart des systèmes sont optimisés pour l’anglais. Des travaux comme [X-Fact](https://github.com/guofei9987/x-fact) tentent de généraliser.

2. **Vérification en temps réel avec streaming**
   Au lieu d’attendre une réponse complète, vérifier au fur et à mesure que le LLM génère du texte.
   **Exemple** : [Comment les LLMs comprennent le son sans même avoir d’oreilles](https://lelabo.ai/articles/comment-les-ia-comprennent-un-peu-le-son-sans-meme-avoir-d-oreilles--confirme) montre comment traiter des données en continu.

3. **Auto-vérification par les LLMs**
   Des techniques comme **Self-Consistency Checking** (faire générer plusieurs réponses et vérifier la cohérence) ou **Chain-of-Thought** (demander au modèle d’expliquer son raisonnement) améliorent la fiabilité.
   **Limite** : ça reste du "meilleur effort", pas une garantie.

4. **Blockchain pour la traçabilité**
   Stocker les sources et les vérifications dans une blockchain pour garantir l’intégrité.
   **Problème** : la blockchain ne résout pas le problème de la qualité des données, juste leur immuabilité.

### Ce qui serait bien (mais qui n’existe pas encore)
- Un **Wikipedia des faits vérifiés**, mis à jour en temps réel par une communauté de contributeurs + IA.
- Des **LLMs avec mémoire épisodique** : un modèle qui se souvient *vraiment* des infos qu’il a vérifiées par le passé.
- Une **norme universelle pour le fact-checking**, comme le HTTPS pour le web.

## FAQ

**[Comment intégrer la vérification des faits dans mon chatbot existant ?]**
Commencez par ajouter un module de RAG avec des sources fiables, puis intégrez un classifieur de crédibilité en amont. Utilisez des bibliothèques comme LangChain ou Haystack pour simplifier l’intégration. Testez d’abord sur un sous-ensemble de requêtes critiques.

**[Quels sont les meilleurs datasets pour entraîner un modèle de fact-checking ?]**
FEVER, Vitamins, and LIAR sont les plus utilisés. Pour le français, le dataset [DEFACTO](https://gitlab.com/defacto/defacto) est une bonne option. Pensez aussi à créer vos propres datasets avec des claims spécifiques à votre domaine.

**[Pourquoi mon système de vérification a-t-il une latence trop élevée ?]**
La latence vient souvent du RAG (recherche dans les vecteurs) ou des appels externes aux APIs de vérification. Solutions : cachez les résultats fréquents, utilisez des modèles légers pour la vérification, et parallélisez les requêtes. Si c’est toujours trop lent, envisagez une architecture asynchrone.
