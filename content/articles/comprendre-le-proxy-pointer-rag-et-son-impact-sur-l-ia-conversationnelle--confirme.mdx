# Proxy-Pointer RAG : le raisonnement temporel sans la corvée sémantique

On ne va pas se mentir : le RAG classique, c’est un peu comme un bibliothécaire qui classe tous les livres par couleur de couverture. Pratique pour trouver *ce qui ressemble* à votre demande, mais pas pour comprendre *quand* ça s’est produit. Le **Proxy-Pointer RAG** propose une alternative : et si on utilisait des pointeurs temporels au lieu de s’embêter avec des embeddings sémantiques pré-calculés ?

D’après *Towards Data Science*, cette approche permet de gérer des requêtes temporelles complexes sans alourdir l’indexation. Franchement, c’est le genre d’idée qui fait tiquer les puristes du RAG, mais qui pourrait bien sauver vos factures cloud.

---

## Les fondements : pourquoi le RAG classique échoue sur le temps

Le RAG traditionnel repose sur deux piliers :
**La récupération** (retrieval) via des embeddings sémantiques
**La génération** (generation) via un LLM qui synthétise le contexte récupéré

Problème : la sémantique, c’est statique. Si votre base de connaissances évolue dans le temps, vos embeddings aussi. Et là, c’est la galère :
- Réindexation coûteuse à chaque mise à jour
- Latence qui explose quand la base grossit
- Impossible de répondre à *"Qu’est-ce qui a changé entre hier et aujourd’hui ?"* sans bidouiller des prompts magiques

Le Proxy-Pointer RAG, lui, remplace les embeddings par des **pointeurs temporels**. Imaginez un système de fichiers où chaque document a une étiquette *"créé le 15/05, modifié le 20/05"*. Au lieu de comparer des vecteurs, on compare des timestamps. Simple, non ?

**Mais attention** : ça ne marche que si vos données sont *naturellement temporelles*. Un historique de commits Git ? Parfait. Une base de connaissances statiques façon Wikipedia ? Bonne chance avec ça.

## Architecture : comment ça marche sous le capot

### 1. Indexation : des pointeurs, pas des vecteurs

Au lieu d’embeddings, on stocke :
- **Un identifiant unique** pour chaque document (UUID, hash, etc.)
- **Des métadonnées temporelles** : timestamp de création, de modification, de validité
- **Un pointeur vers le contenu brut** (dans un storage type S3, HDFS, etc.)

Exemple en Python (simplifié) :

```python
from dataclasses import dataclass
from typing import Dict, List
import uuid
from datetime import datetime

@dataclass
class TemporalDocument:
    id: str
    content: str
    created_at: datetime
    updated_at: datetime
    valid_until: datetime  # Optionnel : pour les données périmables

class ProxyPointerIndex:
    def __init__(self):
        self.documents: Dict[str, TemporalDocument] = {}
        self.time_to_ids: Dict[datetime, List[str]] = {}  # Mapping timestamp -> liste d'IDs

    def add_document(self, content: str, created_at: datetime, updated_at: datetime):
        doc_id = str(uuid.uuid4())
        self.documents[doc_id] = TemporalDocument(
            id=doc_id,
            content=content,
            created_at=created_at,
            updated_at=updated_at,
            valid_until=datetime.max  # Par défaut, jamais périmé
        )
        # Indexation temporelle
        if created_at not in self.time_to_ids:
            self.time_to_ids[created_at] = []
        self.time_to_ids[created_at].append(doc_id)
```

### 2. Récupération : filtrer par temps, pas par similarité

Pour une requête comme *"Quels sont les changements dans la politique de RGPD entre janvier et mars 2025 ?"*, le système :
1. **Parse la requête** pour extraire les bornes temporelles (`janvier 2025`, `mars 2025`)
2. **Récupère tous les documents** dont `updated_at` est dans cet intervalle
3. **Trie par pertinence temporelle** (le plus récent d’abord, par exemple)

```python
def retrieve_by_time_range(self, start: datetime, end: datetime) -> List[TemporalDocument]:
    relevant_ids = []
    for ts, ids in self.time_to_ids.items():
        if start <= ts <= end:
            relevant_ids.extend(ids)
    return [self.documents[doc_id] for doc_id in relevant_ids]
```

**Avantage** : Pas besoin de recalculer des embeddings à chaque mise à jour.
**Inconvénient** : Si votre requête n’a *aucune* composante temporelle, vous êtes bon pour tomber sur un RAG classique en fallback.

### 3. Génération : le LLM fait le reste

Une fois les documents récupérés, le LLM les utilise comme contexte. Rien de révolutionnaire ici, sauf que :
- Le prompt peut inclure des instructions du type *"Prends en compte l’ordre chronologique des documents"*
- On évite les hallucinations liées à des embeddings obsolètes

Exemple de prompt :

```
Tu es un assistant expert en conformité RGPD.
Voici les documents pertinents, triés par date de modification :
1. [Document A] (2025-01-15) : "Nouvelle règle sur le consentement..."
2. [Document B] (2025-02-20) : "Modification de l'article 17..."
3. [Document C] (2025-03-10) : "Clarification sur les sanctions..."

Réponds à la question suivante en tenant compte de l'évolution temporelle :
"Quels sont les changements majeurs dans la politique RGPD entre janvier et mars 2025 ?"
```

## Benchmarks : ça tient la route ?

Les auteurs de l’article *Towards Data Science* ont testé leur approche sur :
- **Un dataset de news** (Reuters, 2010-2023)
- **Un historique de commits Git** (projets open source majeurs)
- **Des logs système** (avec timestamps précis)

Résultats (à prendre avec des pincettes, bien sûr) :

| Métrique               | RAG Classique | Proxy-Pointer RAG | Gain |
|------------------------|---------------|-------------------|------|
| Latence (ms)           | 450           | 80                | -82% |
| Précision temporelle   | 65%           | 92%               | +27% |
| Coût indexation        | Élevé         | Faible            | -90% |
| Mémoire utilisée       | 12 Go         | 2 Go              | -83% |

**À noter** :
- Le gain en précision temporelle est logique : le Proxy-Pointer est *conçu* pour ça.
- La latence et la mémoire s’effondrent car on évite le calcul d’embeddings.
- **Mais** : sur des requêtes purement sémantiques (*"Explique-moi la RGPD"*), le RAG classique reste meilleur.

## Limitations : les pièges à éviter

### 1. Ça ne marche pas sans données temporelles

Si vos documents n’ont pas de timestamps fiables, le Proxy-Pointer RAG devient un RAG aléatoire. **Point.**

### 2. Les requêtes hybrides sont un casse-tête

Une requête comme *"Quels sont les impacts sémantiques des changements RGPD entre 2023 et 2025 ?"* mélange temps *et* sémantique. Là, il faut :
- Soit combiner Proxy-Pointer + RAG classique (et bon courage pour l’orchestration)
- Soit accepter de perdre en précision sur un des deux axes

### 3. La granularité temporelle compte

Si vos timestamps sont à la journée près, impossible de répondre à *"Qu’est-ce qui a changé à 14h30 ?"*. À vous de choisir votre niveau de précision en fonction de vos besoins.

### 4. Les données périmées, un vrai bordel

Si un document a une date de validité (`valid_until`), il faut :
- **Le supprimer** de l’index à échéance
- **Ou le marquer comme obsolète** et l’exclure des requêtes

Sinon, votre LLM va vous sortir des infos dépassées avec la confiance d’un commercial en fin de mois.

## Recherche et évolutions futures

Le Proxy-Pointer RAG est encore jeune, mais quelques pistes se dessinent :

### 1. Hybridation avec du RAG classique

Pourquoi choisir ? Des travaux explorent des systèmes où :
- Le **Proxy-Pointer** gère la dimension temporelle
- Un **RAG vectoriel** gère la dimension sémantique
- Un **classifieur léger** décide quelle approche utiliser en fonction de la requête

### 2. Optimisation des pointeurs

Aujourd’hui, les pointeurs sont basiques (timestamps + IDs). Demain, on pourrait voir :
- Des **pointeurs hiérarchiques** (année > mois > jour > heure)
- Des **pointeurs probabilistes** (pour gérer l’incertitude sur les dates)
- Des **pointeurs compressés** (pour réduire la taille de l’index)

### 3. Intégration avec les graphes de connaissances

Et si on combinait Proxy-Pointer RAG avec un **graphe temporel** ? Imaginez :
- Chaque nœud du graphe a un timestamp
- Les arêtes représentent des relations *et* des intervalles temporels
- Le retrieval devient une **requête temporelle sur graphe**

C’est le genre d’idée qui fait rêver les architectes ML... et cauchemarder les devs.

## FAQ

**[Le Proxy-Pointer RAG remplace-t-il complètement le RAG classique ?]**
Non. Il excelle sur les requêtes temporelles, mais échoue sur les requêtes purement sémantiques. Une approche hybride est souvent nécessaire.

**[Quels sont les cas d’usage idéaux pour le Proxy-Pointer RAG ?]**
Les données naturellement temporelles : logs, historiques de commits, news, données financières, ou tout système où la chronologie est critique.

**[Comment gérer les données sans timestamps dans un Proxy-Pointer RAG ?]**
Soit vous les excluez, soit vous leur attribuez un timestamp par défaut (date de création du dataset, par exemple). Mais attention aux biais introduits.
