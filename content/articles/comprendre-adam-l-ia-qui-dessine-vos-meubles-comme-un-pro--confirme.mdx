# Adam CAD : quand l’IA open source se met au dessin industriel

On ne va pas se mentir : la CAO, c’est un peu comme le yoga. Tout le monde en parle, tout le monde sait que c’est utile, mais au final, seuls les initiés osent s’y mettre. Sauf que là, Adam débarque avec une promesse simple : *"Décris-moi ton meuble, je te le dessine en 3D."* Et le pire, c’est que ça marche. Enfin, presque.

Adam, c’est le nouveau jouet open source de la Y Combinator, un outil text-to-CAD qui veut démocratiser la modélisation 3D. Pas de buzzwords marketing ici, juste du code, des modèles et une ambition : remplacer les heures passées sur SolidWorks par une simple phrase en anglais. On a creusé le repo GitHub, testé les démos et discuté avec les early adopters. Voici ce qu’il faut vraiment retenir.

## Les fondements techniques : comment Adam transforme du texte en géométrie

Adam ne sort pas de nulle part. Le projet s’appuie sur une stack technique solide, avec une approche hybride qui mélange **génération de code** et **modélisation paramétrique**.

### L’architecture sous le capot
Le cœur du système repose sur trois piliers :
- **Un LLM finetuné** : Adam utilise un modèle de langage (non spécifié dans le repo, mais probablement un Mistral ou un Llama sous le capot) entraîné sur des datasets de descriptions techniques et de code CAD. Le modèle comprend des prompts comme *"une table en chêne avec 4 pieds carrés de 50cm de haut"* et les traduit en instructions exécutables.
- **Un moteur de géométrie** : Pour la partie 3D, Adam s’appuie sur **OpenSCAD**, un outil open source de modélisation paramétrique. Contrairement à Blender ou Fusion 360, OpenSCAD ne travaille pas avec des maillages, mais avec du code. Une table ? C’est une série de `cube()`, `cylinder()` et de `translate()`.
- **Un pipeline de post-traitement** : Une fois le code OpenSCAD généré, Adam l’exécute pour produire un fichier `.stl` ou `.step`, prêt à être importé dans n’importe quel logiciel de CAO.

**Exemple concret** : Si vous demandez *"un tabouret en forme de champignon avec un pied central et un siège circulaire"*, Adam va :
1. Analyser la requête avec le LLM pour en extraire les paramètres (forme, dimensions, matériaux implicites).
2. Générer un script OpenSCAD du style :
```openscad
// Pied central
cylinder(h=40, r=5, center=true);
translate([0,0,20])
// Siège circulaire
cylinder(h=3, r=20, center=true);
```
3. Exécuter ce script pour produire un modèle 3D.

Franchement, c’est malin. Mais est-ce que ça tient la route en prod ?

## Implémentation : du PoC au vrai outil

Adam est encore en version alpha, mais le code est déjà disponible sur [GitHub](https://github.com/Adam-CAD/CADAM). Voici ce qu’on a pu observer en fouillant dans les entrailles du projet.

### Stack technique
- **Frontend** : Une interface web en React (Next.js) qui permet de saisir du texte et de visualiser le résultat en temps réel via Three.js.
- **Backend** : Un serveur FastAPI qui gère les requêtes, appelle le LLM (via une API locale ou distante) et génère le code OpenSCAD.
- **Exécution** : Le code OpenSCAD est exécuté dans un conteneur Docker pour éviter les dépendances système. Une bonne pratique, surtout quand on sait que OpenSCAD peut être capricieux selon les OS.

**Point fort** : L’intégration avec OpenSCAD permet une **modélisation paramétrique précise**, bien plus adaptée à la CAO que les approches purement mesh-based (comme celles de [NVIDIA avec Kaolin](https://github.com/NVIDIAGameWorks/kaolin)). Pour les ingénieurs ML, c’est un vrai plus : pas besoin de gérer des maillages complexes, tout est défini par des équations.

**Point faible** : OpenSCAD a ses limites. Pas de modélisation organique (bonjour les formes libres), et une courbe d’apprentissage pour les non-initiés. Adam hérite de ces contraintes.

### Optimisations notables
- **Cache des prompts** : Adam stocke les requêtes fréquentes pour éviter de régénérer le même code OpenSCAD à chaque fois. Un gain de temps non négligeable.
- **Validation des paramètres** : Le LLM est guidé par des *few-shot examples* pour éviter les erreurs de syntaxe dans le code généré. Par exemple, si vous demandez *"un cube de 10m de côté"*, Adam va corriger en *"1000mm"* (OpenSCAD travaille en mm par défaut).
- **Export multi-format** : En plus du `.stl`, Adam peut exporter en `.step` (pour la CAO industrielle) ou en `.gltf` (pour le web).

## Limitations : où Adam coince (pour l’instant)

### 1. La précision des prompts
Adam dépend **beaucoup** de la qualité de votre description. *"Un meuble"* ne suffira pas. Il faut être précis : dimensions, matériaux, assemblages. Sinon, bonjour les surprises.

**Exemple** : *"Une chaise"* → Adam génère un cube avec 4 pieds. *"Une chaise en bois avec dossier incliné à 15°, hauteur 80cm"* → là, on commence à avoir quelque chose d’utilisable.

### 2. Les formes organiques
OpenSCAD n’est pas fait pour ça. Si vous voulez un vase aux courbes sinueuses, Adam va vous proposer un cylindre avec des trous. Point. Pour ça, il faudra attendre une intégration avec Blender ou une approche par *signed distance fields* (SDF).

### 3. Les contraintes industrielles
Adam ne gère pas (encore) les **contraintes mécaniques** (épaisseur minimale, tolérences d’usinage, etc.). Pour de la prototypage rapide, c’est parfait. Pour de la production, il faudra repasser par un logiciel pro.

### 4. La scalabilité
Le projet est encore jeune. Pas de support pour les **assemblages complexes** (ex : un moteur avec 50 pièces), ni de gestion des **hiérarchies de composants**.

## Recherche et évolutions futures

Adam n’est qu’un début. Voici ce qui pourrait arriver dans les prochains mois, d’après les discussions avec l’équipe et la roadmap GitHub :

### 1. Intégration avec d’autres moteurs CAD
L’équipe travaille sur un **backend modulaire** pour supporter d’autres outils que OpenSCAD :
- **FreeCAD** (pour plus de flexibilité)
- **Blender** (pour les formes organiques)
- **Onshape** (pour le cloud)

### 2. Amélioration du LLM
Aujourd’hui, Adam utilise un modèle générique. Demain, il pourrait y avoir :
- Un **finetuning spécifique** sur des datasets de manuels techniques (ex : normes ISO pour les vis, écrous, etc.).
- Une **intégration avec des RAG** (Retrieval-Augmented Generation) pour récupérer des templates de pièces standardisées.

### 3. Collaboration et versioning
Un système de **git pour la CAO** est en discussion. Imaginez : vous poussez une modification sur un modèle 3D, et Adam génère automatiquement le diff en code OpenSCAD. Pratique pour les équipes.

### 4. Plugins pour les pros
Des extensions pour **SolidWorks** ou **Fusion 360** sont envisagées, pour permettre aux ingénieurs de générer des pièces via Adam, puis de les importer directement dans leur logiciel habituel.

Parce que Adam, c’est bien plus qu’un outil de CAO. C’est un **cas d’usage concret de LLM + code generation** avec des contraintes industrielles. Et ça, c’est rare.

## FAQ

**[Adam peut-il remplacer SolidWorks ?]**
Non, pas encore. Adam est parfait pour du prototypage rapide ou des pièces simples, mais il manque encore les fonctionnalités avancées (simulation, usinage, gestion de projets) des logiciels pros. Pour l’instant, c’est un complément, pas un remplaçant.

**[Faut-il savoir coder pour utiliser Adam ?]**
Non. L’interface textuelle est conçue pour être accessible à tous. En revanche, comprendre les bases de la modélisation 3D (dimensions, formes) aide à formuler de meilleurs prompts.

**[Adam est-il gratuit ?]**
Oui, le code est open source (licence MIT) et vous pouvez l’héberger vous-même. En revanche, si vous utilisez une API de LLM externe (comme celle de Mistral ou OpenAI), les coûts dépendront de votre fournisseur.
