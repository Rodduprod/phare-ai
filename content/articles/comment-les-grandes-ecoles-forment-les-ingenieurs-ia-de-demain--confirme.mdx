# Construire un agent IA en entretien : ce que les écoles attendent vraiment

Les grandes écoles l’ont compris : aujourd’hui, un CV avec "Python" et "TensorFlow" ne suffit plus. Selon *Les Échos*, les recruteurs demandent désormais aux candidats de **construire un agent IA** pendant l’entretien. Pas un chatbot basique, non. Un vrai agent autonome, capable de prendre des décisions, d’interagir avec des outils, et surtout, de ne pas planter en production.

Franchement, c’est un bon filtre. Parce que si vous savez coder un agent IA qui tient la route, vous savez aussi :
- **Orchestrer des LLM** sans les laisser divaguer
- **Gérer des boucles de feedback** sans créer de *hallucinations en cascade*
- **Optimiser des coûts d’inférence** sans faire exploser le budget cloud

Bref, vous n’êtes pas un script kiddie qui colle du code trouvé sur Stack Overflow.

## Les fondements techniques : ce qu’un agent IA doit savoir faire

Un agent IA, c’est bien plus qu’un LLM avec une API. C’est un système qui :
1. **Perçoit** son environnement (via des inputs structurés ou non)
2. **Raisonne** (avec ou sans mémoire)
3. **Agit** (en appelant des outils, en prenant des décisions)
4. **Apprend** (ou du moins, s’adapte)

### L’architecture de base : LLM + Outils + Mémoire

Prenons un cas concret : un agent qui gère des tickets de support client.
- **LLM** : Le cerveau. Un modèle comme Mistral 7B ou Llama 3, finetuné ou non.
- **Outils** : Les bras. Des fonctions pour lire une base de connaissances, envoyer un email, ou mettre à jour un ticket dans Jira.
- **Mémoire** : Le carnet de notes. Un système de RAG (Retrieval-Augmented Generation) ou une base vectorielle pour se souvenir des conversations passées.

```python
from langchain.agents import AgentExecutor, Tool
from langchain_core.messages import HumanMessage
from langchain_core.prompts import ChatPromptTemplate

# Définition des outils
def rechercher_docs(query: str) -> str:
    # Appel à une base de connaissances
    return f"Résultat pour {query}"

def envoyer_email(destinataire: str, sujet: str, corps: str) -> str:
    # Envoi d'email via SMTP ou API
    return f"Email envoyé à {destinataire}"

tools = [
    Tool.from_function(rechercher_docs, name="rechercher_docs", description="Recherche dans la base de connaissances"),
    Tool.from_function(envoyer_email, name="envoyer_email", description="Envoie un email")
]

# Prompt de l'agent
prompt = ChatPromptTemplate.from_template(
    "Tu es un agent de support. Utilise les outils disponibles pour répondre à la demande. {input}"
)

# Création de l'agent (simplifié)
agent = prompt | llm.bind_tools(tools)
executor = AgentExecutor(agent=agent, tools=tools)
```

On ne va pas se mentir : ce code, c’est du LangChain 101. **Le vrai défi, c’est de le faire marcher en conditions réelles.**

## Implémentation : les pièges qui font la différence

### 1. La gestion des erreurs (ou comment éviter que votre agent ne crash)

Un agent IA qui appelle des outils, c’est comme un enfant avec un marteau : ça finit mal si on ne surveille pas.

- **Problème** : Un outil plante, le LLM génère du JSON invalide, ou pire, entre dans une boucle infinie.
- **Solution** :
  - **Timeouts** sur les appels d’outils
  - **Validation des outputs** avant exécution
  - **Fallbacks** (ex : si la recherche échoue, essayer une autre source)

```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=4, max=10))
def safe_rechercher_docs(query: str) -> str:
    try:
        return rechercher_docs(query)
    except Exception as e:
        print(f"Erreur : {e}")
        raise
```

### 2. L’optimisation des coûts (parce que personne ne veut payer 10€ par requête)

Un agent qui appelle GPT-4 pour chaque petite tâche, c’est comme utiliser un marteau-piqueur pour enfoncer un clou.

- **Stratégie 1** : **Routing de modèles**. Utiliser un petit modèle (ex : Mistral 7B) pour les tâches simples, et un gros (ex : GPT-4) seulement si nécessaire.
- **Stratégie 2** : **Caching agressif**. Stocker les réponses fréquentes dans une base Redis.
- **Stratégie 3** : **Distillation**. Entraîner un petit modèle à imiter le comportement de l’agent complet.

D’après [notre analyse des coûts d’inférence](https://www.lelabo.ai/articles/pourquoi-les-petits-modeles-ia-sur-votre-bureau-menacent-les-geants-du-cloud--confirme), un agent mal optimisé peut coûter **10 à 100 fois plus cher** qu’un agent bien conçu.

### 3. La mémoire (ou comment éviter que votre agent oublie tout)

Un agent sans mémoire, c’est comme un poisson rouge : il oublie tout après 3 secondes.

- **Mémoire à court terme** : Le contexte de la conversation (via les messages précédents).
- **Mémoire à long terme** : Une base de données vectorielle (ex : Pinecone, Weaviate) pour stocker les informations importantes.
- **Mémoire procédurale** : Des règles métiers (ex : "Si le client est Premium, priorité absolue").

```python
from langchain.memory import ConversationBufferMemory

memory = ConversationBufferMemory(
    return_messages=True,
    memory_key="chat_history"
)

# Intégration dans l'agent
agent_with_memory = {
    "input": lambda x: x["input"],
    "chat_history": lambda x: memory.load_memory_variables(x)["history"],
} | prompt | llm.bind_tools(tools)
```

## Benchmarks : comment évaluer un agent IA ?

Un agent IA, ça se teste. Pas avec des "Hello World", mais avec des **scénarios réels**.

### 1. Senior SWE-Bench : le test qui sépare les pros des amateurs

[Senior SWE-Bench](https://www.lelabo.ai/articles/senior-swe-bench-comment-on-teste-enfin-les-ia-comme-des-vrais-ingenieurs--amateur) est un benchmark qui évalue la capacité d’un agent à :
- **Comprendre un repo GitHub**
- **Corriger des bugs**
- **Ajouter des features**

Résultat moyen aujourd’hui :
- **GPT-4** : ~15% de succès
- **Claude 3** : ~12%
- **Un humain junior** : ~30%

Oui, **votre stagiaire code mieux que GPT-4**. Mais un agent bien conçu peut monter à **40-50%** en combinant :
- **Recherche de code** (via RAG sur le repo)
- **Exécution de tests** (pour valider les changements)
- **Itérations** (si le premier essai échoue)

### 2. AgentBench : le benchmark tout-en-un

[AgentBench](https://agentbench.ai/) teste :
- **Raisonnement** (problèmes mathématiques, logique)
- **Utilisation d’outils** (appels API, manipulation de fichiers)
- **Mémoire** (suivi de conversations longues)

Exemple de tâche :
> *"Trouve le fichier le plus gros dans /home/user, puis envoie-le par email à alice@example.com."*

Un bon agent doit :
1. Lister les fichiers (`ls -lh /home/user`)
2. Trouver le plus gros (`sort -k5 -hr`)
3. Envoyer le fichier (`scp` + `sendmail`)

**Score moyen** :
- **AutoGen** : 65%
- **LangChain** : 55%
- **Un script bash** : 100% (mais bon, on triche un peu)

## Limitations : ce que personne ne vous dit

### 1. Les agents IA sont lents (et ça, c’est un problème)

Un agent qui doit :
1. Lire un document
2. Faire 3 appels API
3. Générer une réponse
... peut mettre **10 à 30 secondes** par requête.

**Solutions** :
- **Parallélisation** des appels d’outils
- **Pré-chargement** des données fréquentes
- **Streaming** des réponses (pour donner l’illusion de rapidité)

### 2. Ils mentent (et pas qu’un peu)

Un agent IA, c’est comme un politicien : il invente des chiffres si ça arrange son raisonnement.

Exemple réel (testé en interne) :
> *"Quelle est la capital de la France ?"*
> **Agent** : *"La capitale de la France est Lyon, car c’est la ville la plus importante économiquement."*

**Pourquoi ?** Parce que le LLM a été entraîné sur des données où "Lyon" apparaît souvent avec "économie", et il a **halluciné** une réponse.

**Solutions** :
- **Validation externe** (ex : vérifier sur Wikipedia avant de répondre)
- **Contraintes strictes** (ex : "Ne réponds que si tu es sûr à 100%")
- **Fallbacks** (ex : "Je ne sais pas" au lieu d’inventer)

### 3. Ils coûtent une blinde

Un agent qui tourne 24/7 avec :
- 1 appel à GPT-4 par minute
- 10 appels à des APIs externes
- Stockage dans Pinecone

**Coût mensuel estimé** : **5 000 à 20 000€**

**Comment réduire la facture ?**
- Utiliser des **modèles open source** (Mistral, Llama)
- **Cacher les réponses** fréquentes
- **Optimiser les prompts** (moins de tokens = moins cher)

## Recherche & évolutions futures : vers où on va ?

### 1. Les agents multi-modaux (parce que le texte, c’est has been)

Demain, un agent devra :
- **Voir** (images, vidéos)
- **Entendre** (audio, voix)
- **Agir** (contrôle de robots, IoT)

Exemple : Un agent qui :
1. **Analyse une photo** de panne industrielle
2. **Écoute** les bruits de la machine
3. **Propose une solution** (et envoie un technicien si nécessaire)

**Technos à surveiller** :
- **LLMs vision** (ex : LLaVA, Flamingo)
- **ASR (Automatic Speech Recognition)** (ex : Whisper, Wav2Vec)
- **Robotique + LLM** (ex : [NVIDIA ProRL Agent](https://www.lelabo.ai/articles/nvidia-lance-prorl-agent-l-usine-a-ia-qui-apprend-toute-seule--confirme))

### 2. Les agents collaboratifs (l’IA en équipe)

Un seul agent, c’est bien. **Une équipe d’agents**, c’est mieux.

Exemple :
- **Agent 1** : Spécialiste du code (corrige les bugs)
- **Agent 2** : Spécialiste des tests (vérifie que ça marche)
- **Agent 3** : Spécialiste du déploiement (push sur GitHub)

**Architecture** :
```python
from crewai import Agent, Task, Crew

# Définition des agents
codeur = Agent(role="Développeur Senior", goal="Corriger les bugs", backstory="...", llm=llm)
testeur = Agent(role="QA Engineer", goal="Tester le code", backstory="...", llm=llm)

# Tâches
fix_bug = Task(description="Corriger le bug dans le repo X", agent=codeur)
test_fix = Task(description="Tester la correction", agent=testeur, context=[fix_bug])

# Équipe
crew = Crew(agents=[codeur, testeur], tasks=[fix_bug, test_fix])
result = crew.kickoff()
```

**Avantages** :
- **Spécialisation** (chaque agent fait ce qu’il sait faire)
- **Redondance** (si un agent échoue, un autre prend le relais)
- **Scalabilité** (on peut ajouter des agents sans tout recoder)

### 3. Les agents avec mémoire à long terme (la vraie)

Aujourd’hui, la mémoire des agents, c’est comme un Post-it : ça tient 2 minutes, puis ça s’envole.

Demain, on veut :
- **Mémoire épisodique** (se souvenir des événements passés)
- **Mémoire sémantique** (comprendre les concepts)
- **Mémoire procédurale** (savoir comment faire les choses)

**Technos prometteuses** :
- **Graphes de connaissances** (ex : Neo4j)
- **Mémoire bio-inspirée** (ex : [Hippo](https://www.lelabo.ai/articles/comment-hippo-donne-une-memoire-d-elephant-aux-agents-ia-sans-les-droguer--confirme))
- **Apprentissage continu** (fine-tuning en temps réel)

## FAQ

**[Comment évaluer un candidat qui construit un agent IA en entretien ?]**
Regardez trois choses : la **robustesse** (gère-t-il les erreurs ?), l’**optimisation** (est-ce que ça coûte 10€ par requête ?), et la **modularité** (peut-on ajouter un outil sans tout recoder ?). Un bon candidat sait aussi expliquer **pourquoi** il a fait certains choix techniques.

**[Quels outils utiliser pour construire un agent IA rapidement ?]**
LangChain ou CrewAI pour l’orchestration, FastAPI pour l’API, Pinecone ou Weaviate pour la mémoire, et des LLM open source (Mistral, Llama) pour réduire les coûts. Évitez les usines à gaz : un agent simple mais fonctionnel vaut mieux qu’un truc complexe qui plante.

**[Quels sont les principaux défis pour déployer un agent IA en production ?]**
La **latence** (les appels LLM sont lents), la **fiabilité** (un agent qui hallucine, c’est un risque business), et le **coût** (un agent mal optimisé peut coûter une fortune). Ajoutez à ça la **sécurité** (un agent avec accès à vos APIs, c’est un risque de fuite de données).
