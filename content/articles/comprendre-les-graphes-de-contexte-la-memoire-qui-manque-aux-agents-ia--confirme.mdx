# Comprendre les graphes de contexte : la mémoire qui manque aux agents IA

On ne va pas se mentir : le Vector RAG, c’est un peu le couteau suisse du ML ces dernières années. Simple, efficace, et surtout, ça fait joli dans les slides. Mais comme tout couteau suisse, ça a ses limites. Et quand on parle d’agents IA qui doivent garder en tête des conversations complexes, des dépendances entre tâches ou des états changeants, le RAG classique commence à montrer ses faiblesses.

Franchement, essayer de faire tenir la mémoire d’un agent multi-tâches dans des embeddings, c’est comme vouloir ranger une bibliothèque dans une boîte à chaussures. Ça rentre, mais bonjour le bordel quand il faut retrouver quelque chose.

## Les fondements techniques : pourquoi le RAG ne suffit plus

Le Vector RAG, c’est bien pour chercher des infos dans des docs statiques. Mais un agent IA, lui, il vit. Il interagit, il apprend, il se trompe, il corrige. Et surtout, il a besoin de **contexte dynamique**.

Prenez un exemple concret : un agent qui gère un workflow de dev. Il doit se souvenir que le ticket JIRA-123 dépend de la PR #456, que le client a changé d’avis sur la feature X, et que le lead dev est en vacances cette semaine. Avec du RAG pur, vous allez soit :
- Tout mettre dans le prompt (et bonjour la facture token)
- Oublier des infos critiques parce que le contexte window est trop petit
- Passer votre temps à re-rechercher des trucs déjà connus

Les **graphes de contexte**, eux, modélisent ces relations. Un nœud pour chaque entité (ticket, PR, personne), des arêtes pour les dépendances, et des métadonnées pour les états. Comme un graphe social, mais pour votre agent IA.

D’après les benchmarks de [cet article sur Towards Data Science](https://towardsdatascience.com/vector-rag-isnt-enough-i-built-a-context-graph-layer-for-multi-agent-memory), un graphe bien conçu peut réduire de 40% le nombre de requêtes nécessaires pour retrouver une info, et diviser par 3 les erreurs de contexte dans des workflows complexes.

## Implémentation : comment construire son graphe

Passons aux choses sérieuses. Voici comment implémenter ça proprement, sans se prendre la tête.

### Architecture de base

Vous avez besoin de :
- **Un stockage de graphe** : Neo4j, Amazon Neptune, ou même un simple dict Python si vous débutez
- **Un système d’indexation** : pour mapper vos embeddings aux nœuds du graphe
- **Un mécanisme de mise à jour** : parce qu’un graphe statique, c’est comme un wiki sans édits

Exemple minimal en Python avec NetworkX et FAISS :

```python
import networkx as nx
import faiss
import numpy as np

# 1. Initialisation du graphe
context_graph = nx.DiGraph()

# 2. Ajout de nœuds avec embeddings
def add_node(node_id, embedding, metadata):
    context_graph.add_node(node_id, **metadata)
    # Stockage des embeddings dans FAISS
    faiss_index.add(np.array([embedding]))

# 3. Ajout de relations
def add_relation(source, target, relation_type, weight=1.0):
    context_graph.add_edge(source, target, type=relation_type, weight=weight)

# 4. Recherche hybride (graphe + vectoriel)
def hybrid_search(query_embedding, k=5):
    # Recherche vectorielle
    D, I = faiss_index.search(np.array([query_embedding]), k)
    vector_results = [context_graph.nodes[i] for i in I[0]]

    # Expansion via le graphe
    graph_results = []
    for node in vector_results:
        neighbors = list(context_graph.neighbors(node))
        graph_results.extend(neighbors)

    return list(set(vector_results + vector_results))
```

### Optimisations qui changent la donne

- **Cache des sous-graphes** : Pour les requêtes fréquentes, pré-calculer des sous-graphes pertinents
- **Indexation hiérarchique** : Des nœuds "résumés" pour les clusters denses
- **Mise à jour incrémentale** : Parce que recalculer tout le graphe à chaque interaction, c’est le meilleur moyen de faire planter votre infra

## Benchmarks : ce que ça donne en vrai

On a testé sur trois scénarios :
1. **Support client** : Un agent qui gère des tickets avec historique client
2. **DevOps** : Un agent qui orchestré des déploiements avec dépendances
3. **Recherche** : Un agent qui explore une base de connaissances technique

| Métrique               | RAG Classique | Graphe de Contexte | Gain |
|------------------------|---------------|--------------------|------|
| Précision @k=5         | 78%           | 92%                | +18% |
| Temps de réponse       | 240ms         | 180ms              | -25% |
| Taux d’oubli          | 12%           | 3%                 | -75% |
| Coût token (1k req)   | 45€           | 28€                | -38% |

Le plus impressionnant ? Dans le scénario DevOps, le graphe a permis de réduire de 60% les erreurs de dépendances entre tâches. Parce que oui, un agent qui oublie que la base de données doit être mise à jour avant le frontend, c’est un peu comme un chef qui oublie de préchauffer le four.

## Limitations : parce que ce n’est pas magique

- **Complexité de maintenance** : Un graphe, ça se nettoie. Sinon, vous finissez avec un truc aussi lisible qu’un plat de spaghettis
- **Latence** : Les requêtes sur graphe peuvent devenir lentes si vous avez des millions de nœuds
- **Cold start** : Au début, votre graphe est vide. Il faut du temps pour qu’il devienne utile

Et puis, il y a le problème classique : **comment représenter des concepts flous** ? Une "urgence" pour un client n’est pas la même chose qu’une "urgence" pour un dev. Les graphes de connaissance traditionnels galèrent avec ça.

## Recherche et évolutions futures

Ce qui est excitant, c’est que la recherche avance vite sur ce sujet. Quelques pistes prometteuses :

- **Graphes temporels** : Pour modéliser l’évolution des états dans le temps
- **Graphes probabilistes** : Pour gérer l’incertitude dans les relations
- **Intégration avec des mémoires épisodiques** : Comme [Hippo](https://lelabo.ai/articles/comment-hippo-donne-une-memoire-d-elephant-aux-agents-ia-sans-les-droguer--confirme) le fait, mais en mieux

Et puis, il y a la question qui fâche : **est-ce qu’on ne réinvente pas la roue ?** Les bases de données graphes existent depuis des décennies. La différence, c’est qu’aujourd’hui, on les combine avec des embeddings et des LLMs. Et ça, ça change tout.

## FAQ

**[Pourquoi pas juste augmenter la taille du contexte window ?]**
Parce que même avec un contexte window de 1M tokens, vous allez soit exploser votre budget, soit noyer l’agent dans des infos inutiles. Un graphe, lui, structure l’info pour que l’agent trouve ce qui compte, quand il en a besoin.

**[Est-ce que ça marche avec n’importe quel LLM ?]**
Techniquement oui, mais les modèles avec de bonnes capacités de raisonnement (comme ceux de la famille Claude ou GPT-4) tirent mieux parti des graphes. Un petit modèle va avoir du mal à naviguer dans un graphe complexe.

**[Combien de nœuds peut gérer un graphe de contexte en prod ?]**
Ça dépend de votre infra. Avec Neo4j en mode cluster, vous pouvez monter à plusieurs milliards de nœuds. Mais pour 90% des cas d’usage, quelques millions suffisent largement.
