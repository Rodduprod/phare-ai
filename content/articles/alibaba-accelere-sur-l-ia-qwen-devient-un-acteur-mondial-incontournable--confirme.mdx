# Qwen 3.5 : dissection technique de l'offensive IA d'Alibaba en 2025

Alors qu'Alibaba dément fermement les rumeurs de démissions collectives au sein de son équipe Qwen — relayées par Yahoo Finance début juillet 2025 — et affiche une feuille de route d'expansion accélérée, il est temps de dépasser la narration corporate pour examiner ce que les architectures Qwen impliquent réellement pour les praticiens ML. Voici ce que les benchmarks publics, les poids open-source et les papiers techniques permettent de disséquer.

## Fondements techniques : l'architecture MoE au cœur de Qwen

La famille Qwen repose désormais presque exclusivement sur une architecture **Mixture-of-Experts (MoE)** pour ses variantes grandes échelles, et un backbone dense pour les modèles embarqués (Qwen2.5-0.5B à 7B). L'évolution vers Qwen3 a introduit plusieurs changements structurels significatifs.

### Routing granulaire et experts spécialisés

Qwen3-235B-A22B active environ **22 milliards de paramètres par token** sur 235 milliards totaux. Le mécanisme de routing utilise un **top-K expert selection** avec K=8 parmi 128 experts par couche, combiné à une normalisation par softmax sur les logits de routing — une approche proche de DeepSeek-V2 mais avec des ajustements sur la régularisation du load balancing.

```python
# Schéma simplifié du routing MoE Qwen3
import torch
import torch.nn.functional as F

class QwenMoERouter(torch.nn.Module):
    def __init__(self, hidden_dim, num_experts=128, top_k=8):
        super().__init__()
        self.gate = torch.nn.Linear(hidden_dim, num_experts, bias=False)
        self.top_k = top_k
        self.num_experts = num_experts

    def forward(self, x):
        # x: [batch, seq_len, hidden_dim]
        logits = self.gate(x)  # [batch, seq_len, num_experts]
        
        # Auxiliary loss pour load balancing
        probs = F.softmax(logits, dim=-1)
        top_k_probs, top_k_indices = torch.topk(probs, self.top_k, dim=-1)
        
        # Normalisation des scores sélectionnés
        top_k_probs = top_k_probs / top_k_probs.sum(dim=-1, keepdim=True)
        
        return top_k_probs, top_k_indices, probs

Le coût de l'auxiliary loss de load balancing reste un compromis délicat : un coefficient trop élevé homogénéise l'utilisation des experts et détruit la spécialisation, trop faible et quelques experts saturent systématiquement.

### Attention : GQA et contexte long

Qwen3 utilise du **Grouped Query Attention (GQA)** avec un ratio de grouping variable selon la taille du modèle. Pour Qwen3-72B, on observe 8 groupes de Q pour 8 têtes KV — un rapport 1:1 qui s'approche du MHA classique pour préserver la qualité sur les longues séquences. Les variantes plus petites descendent à 4 têtes KV pour 28 têtes Q, réduisant significativement l'empreinte mémoire du KV cache.

La gestion du contexte long (jusqu'à **128K tokens** sur Qwen3-72B) repose sur **YaRN** (Yet another RoPE extensioN) pour l'interpolation des positions, avec une window size configurée à 4096 pour l'attention locale dans les variantes hybrides sparse/dense.

### Thinking mode vs. non-thinking

Contrairement à d'autres architectures de raisonnement qui maintiennent deux modèles séparés, Qwen3 intègre un **mode thinking commutable** via un token de contrôle (`/think` et `/no_think`). En mode thinking, le modèle génère une chaîne de raisonnement interne avant la réponse finale — implémentée comme une génération conditionnelle plutôt qu'un véritable module séparé. Cela simplifie le déploiement mais impose un overhead de latence non négligeable en mode raisonnement.

Le `chunked-prefill` est particulièrement important pour les requêtes à contexte long : il évite les pics de mémoire lors du prefill et améliore le throughput global de 15-25% selon la distribution des longueurs de séquences en entrée.

### KV Cache et mémoire

Pour Qwen3-72B en BF16, le KV cache représente :
- **Par token** : 2 × num_layers × num_kv_heads × head_dim × 2 bytes = 2 × 80 × 8 × 128 × 2 = ~327 KB
- Pour un batch de 32 séquences à 8K tokens : ~83 GB uniquement pour le KV cache

Le passage en GQA avec 8 têtes KV (vs 64 en MHA classique) divise ce coût par 8, ce qui explique en grande partie pourquoi les variantes récentes de Qwen peuvent tenir des contextes longs sur du matériel raisonnable. ### Résultats sur les benchmarks standards

D'après les évaluations publiées par Alibaba et vérifiées partiellement par la communauté sur Open LLM Leaderboard :

|--------|------|-----------|------|-------|------|

| Llama-3.1-405B | 85.2 | 89.0 | 73.8 | 96.8 | 50.7 |

Les chiffres sur MATH et GPQA (Graduate-Level Google-Proof Q&A) sont particulièrement notables : Qwen3 surpasse GPT-4o sur les benchmarks de raisonnement scientifique, un domaine où les modèles chinois ont historiquement rattrapé leur retard via des datasets de préentraînement mathématiques intensifs.

### Nuances critiques

Ces benchmarks appellent plusieurs mises en garde importantes :

1. **Contamination des données** : Qwen3 ayant été entraîné après la publication de ces benchmarks, le risque de contamination reste non nul malgré les précautions déclarées.
2. **Mode thinking activé** : une partie des scores élevés sur MATH et GPQA est obtenue avec le thinking mode, ce qui multiplie les tokens générés (et la latence) par un facteur 3-10x.
3. **Variance inter-runs** : sur HumanEval notamment, des écarts de ±2-3 points sont fréquents selon les paramètres de sampling.

## Limitations : les angles morts de l'architecture

### Latence en mode raisonnement

Le thinking mode de Qwen3 génère en moyenne 2000-8000 tokens de raisonnement interne avant la réponse finale. Sur une infrastructure A100 80GB, cela se traduit par des latences de 15-60 secondes pour des problèmes complexes — incompatible avec de nombreux use cases temps réel. Comparer avec des architectures comme [MCP et ses patterns de déploiement](/articles/mcp-protocole-connecte-ia-monde-reel) qui permettent de déléguer le raisonnement à des agents asynchrones.

### Instabilité du routing MoE

En pratique, les modèles MoE souffrent de **routing collapse** lors des fine-tunings agressifs : certains experts sont systématiquement sous-exploités, réduisant le modèle à sa capacité effective d'un modèle dense de taille bien inférieure. Qwen3 utilise un expert dropout pendant l'entraînement (taux 10%) pour mitiger ce phénomène, mais le problème persiste lors des adaptations downstream sans precaution.

### Multilingue déséquilibré

Malgré les affirmations d'Alibaba sur le support de 119 langues, les performances chutent significativement en dehors du mandarin, de l'anglais, et d'une poignée de langues européennes majeures. Sur des benchmarks multilingues comme FLORES-200, les langues africaines et d'Asie du Sud-Est montrent des dégradations de 15-30 points par rapport au baseline anglais.

## Recherche & évolutions futures

### Spécialisation par domaine avec LoRA MoE

La direction de recherche la plus prometteuse pour Qwen3 en entreprise est l'adaptation via **MoE LoRA** : plutôt que d'appliquer des adaptateurs LoRA uniformément, on entraîne des adaptateurs spécifiques par expert, permettant une spécialisation domaine sans modifier le routing. Des travaux récents (MoLoRA, MixLoRA) montrent des gains de 8-12 points sur des tâches métier spécialisées avec seulement 50-100M de paramètres additionnels.

### Distillation vers les variantes edge

Alibaba investit visiblement dans la chaîne de distillation Qwen3-235B → Qwen3-1.7B, avec un focus sur le transfert des capacités de raisonnement. Les résultats préliminaires sur Qwen3-4B sont surprenants : avec thinking mode, ce modèle compact surpasse Qwen2.5-72B sur plusieurs benchmarks de raisonnement, suggérant que la distillation de "traces de pensée" est particulièrement efficace.

### Vers une architecture hybride dense/sparse

Les prochaines itérations devraient selon toute vraisemblance combiner des couches denses pour le bas du réseau (traitement des features bas-niveau) avec des couches MoE pour le raisonnement de haut niveau — une architecture que certains appellent **hybrid MoE** et que DeepSeek a également explorée. Cette approche réduit le routing overhead pour les couches où la spécialisation apporte peu de valeur.

## Conclusion

L'expansion Qwen d'Alibaba n'est pas qu'un mouvement de communication corporate : les choix architecturaux (MoE 128 experts, GQA, thinking mode commutable, YaRN pour les longs contextes) reflètent une ingénierie sérieuse, compétitive avec l'état de l'art occidental sur plusieurs axes. Pour les équipes ML françaises, l'open-source de la famille Qwen représente une alternative crédible à Llama pour les applications nécessitant de fortes capacités de raisonnement mathématique ou scientifique, avec un écosystème tooling (vLLM, Ollama, LMStudio) désormais mature. Les limitations réelles — latence du thinking mode, instabilité MoE en fine-tuning, multilinguisme déséquilibré — méritent d'être mesurées sur vos propres distributions de données avant tout déploiement en production.
