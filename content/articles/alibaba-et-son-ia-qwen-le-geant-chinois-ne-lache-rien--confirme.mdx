# Qwen 3.5 et au-delà : l'architecture MoE d'Alibaba sous le microscope

Alors que Yahoo Finance rapportait récemment l'expansion agressive du programme Qwen d'Alibaba — tout en démentant les rumeurs de démissions collectives au sein de l'équipe R&D — la vraie question pour un ingénieur ML n'est pas de nature RH. Elle est technique : que valent réellement les choix architecturaux de Qwen face aux standards actuels de l'industrie, et où se situent les prochains paris d'Alibaba ?

## Fondements techniques : la philosophie MoE de Qwen

La famille Qwen repose sur une architecture **Mixture-of-Experts (MoE)** pour ses modèles de grande taille, un choix désormais presque canonique mais dont les détails d'implémentation varient sensiblement d'un lab à l'autre.

Dans Qwen3-235B-A22B (le flagship actuel), on trouve :
- **235 milliards de paramètres totaux**, dont **22 milliards activés par token**
- **128 experts** au total, avec un routage **top-8 par token**
- Des couches d'attention **GQA (Grouped Query Attention)** avec un ratio de 4:1 entre têtes de query et têtes de key/value
- Une fenêtre de contexte de **128K tokens** en natif, extensible via YaRN

Le routage des experts utilise un mécanisme de **load balancing auxiliaire** avec un coefficient de perte additive (typiquement `α = 0.001` dans les configs publiées), évitant l'effondrement sur un sous-ensemble d'experts — problème classique des architectures MoE naïves.

```python
# Schéma simplifié du routage MoE de Qwen3
import torch
import torch.nn.functional as F

class QwenMoERouter(nn.Module):
    def __init__(self, hidden_dim, num_experts, top_k):
        super().__init__()
        self.gate = nn.Linear(hidden_dim, num_experts, bias=False)
        self.top_k = top_k
        self.num_experts = num_experts

    def forward(self, x):
        # x: [batch, seq_len, hidden_dim]
        logits = self.gate(x)  # [batch, seq_len, num_experts]
        scores = F.softmax(logits, dim=-1)
        
        # Sélection top-k
        top_k_scores, top_k_indices = torch.topk(scores, self.top_k, dim=-1)
        top_k_scores = top_k_scores / top_k_scores.sum(dim=-1, keepdim=True)
        
        # Auxiliary load balancing loss
        expert_load = scores.mean(dim=[0, 1])  # fraction par expert
        balance_loss = self.num_experts * (expert_load * scores.mean(dim=[0,1])).sum()
        
        return top_k_indices, top_k_scores, balance_loss

### Thinking Mode : un système de raisonnement hybride

Une des innovations majeures de Qwen3 est son **mode de pensée hybride** (thinking/non-thinking). Le modèle peut être invoqué en mode "extended thinking" via un flag spécifique au prompt system, générant un CoT interne qui n'est pas exposé à l'utilisateur final mais conditionne la réponse finale. C'est architecturalement différent de l'approche o1/o3 d'OpenAI : il ne s'agit pas d'un modèle séparé de reward mais d'un **fine-tuning RLVR (Reinforcement Learning with Verifiable Rewards)** sur le même checkpoint de base.

Selon la documentation technique publiée par Alibaba, l'entraînement RLVR utilise un verifier externe pour les domaines à réponse vérifiable (mathématiques, code, logique formelle), et un **modèle de récompense LLM-as-judge** pour les tâches ouvertes.

## Limitations : les angles morts de l'architecture

### 1. Latence au premier token (TTFT)

Le prix à payer du MoE activé à 22B de paramètres sur 235B est une latence de dispatch non négligeable. Sur un setup 8×H100, le **Time-To-First-Token** pour une séquence de 4K tokens est d'environ **480ms**, contre ~180ms pour un dense de taille équivalente activée (Llama 3.1 70B). Cette asymétrie devient critique pour les applications temps-réel ou les pipelines d'agents à haute fréquence.

### 2. Dégradation sur les longues fenêtres de contexte

Malgré les 128K tokens annoncés, les tests de "needle-in-a-haystack" publiés par des tiers (Gradient, Latent Space) révèlent une **dégradation significative au-delà de 64K tokens** sur les tâches de retrieval précis. Le mécanisme YaRN utilisé pour l'extension de contexte introduit une variance plus élevée sur les positions distantes.

### 3. Multilingue : l'angle mort non-asiatique

Le corpus d'entraînement de Qwen3 reste fortement déséquilibré en faveur du mandarin et de l'anglais. Les performances sur les langues peu dotées (swahili, vietnamien, langues d'Afrique subsaharienne) restent significativement inférieures aux leaders sur ces niches. Pour les équipes déployant en contexte multilingue européen, la comparaison avec des modèles comme [ERNIE 5.0 et son architecture technique dédiée](/articles/ernie-5-0-le-chatgpt-chinois-qui-conquiert-200-millions-d-utilisateurs--confirme) peut s'avérer pertinente selon les cas d'usage.

### 4. Tool Calling et fiabilité en production

Les évaluations sur Berkeley Function Calling Leaderboard placent Qwen3-235B à **85.6% de succès** sur les appels de fonctions complexes (multi-turn, parallel calls). C'est bon, mais en deçà de GPT-4o (88.1%) et Claude 3.7 Sonnet (87.4%) sur les scénarios à fort emboîtement de fonctions.

## Recherche & évolutions futures

### Qwen VL et la convergence multimodale

La prochaine frontière pour l'équipe Qwen est clairement la **fusion vision-langage native**, au-delà des adaptateurs visuels actuels de Qwen2.5-VL. Selon les publications récentes de l'équipe Tongyi (d'après les preprints ArXiv de Q1 2025), ils explorent une architecture où les **vision tokens sont traités par un sous-ensemble dédié d'experts MoE**, séparés des experts textuels — une approche de spécialisation modale qui réduit les interférences d'entraînement.

### Qwen-Agent et l'orchestration native

L'expansion évoquée par Yahoo Finance concerne aussi l'écosystème **Qwen-Agent** : un framework d'orchestration en Python qui intègre nativement les appels d'outils, la mémoire à long terme et le ReAct loop. L'objectif architectural est de réduire le nombre d'allers-retours LLM nécessaires pour compléter une tâche composée, en préchargeant des sous-routines spécialisées dans le contexte système.

### Distillation et modèles edge

La roadmap publique indique un effort important sur les **modèles Qwen3 distillés** (0.5B, 1.5B, 7B) optimisés pour déploiement sur device. Les versions 7B avec quantisation GPTQ-4bit ciblent explicitement les GPU mobiles NVIDIA Orin et les accélérateurs Apple Neural Engine — un positionnement différent de Llama 3.2 qui vise plutôt le cloud edge.

### Vers un pré-entraînement continu

L'un des paris les plus intéressants d'Alibaba est le passage à un **entraînement continu (continual pre-training)** plutôt que des cycles discrets de release. Cela implique des défis de stabilité du gradient sur des MoE de grande taille (catastrophic forgetting des experts minoritaires), un problème de recherche actif où plusieurs équipes publient en ce moment même sur les stratégies de replay buffer et de regularisation élastique.

## Conclusion technique

L'offensive Qwen n'est pas qu'une manœuvre marketing : les choix architecturaux — routage MoE affiné, RLVR pour le raisonnement, hybridation thinking/non-thinking — reflètent une maturité technique réelle. Les limitations identifiées (TTFT, contexte long, tool calling) sont des vecteurs de travail concrets pour les équipes ML qui envisagent une intégration en production. La prochaine année sera déterminante : si Alibaba réussit la convergence multimodale native et l'entraînement continu à cette échelle, Qwen cessera d'être un challenger pour devenir un standard de référence indépendant.
