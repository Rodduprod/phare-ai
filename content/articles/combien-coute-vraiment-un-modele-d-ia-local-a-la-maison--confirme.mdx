# Combien coûte vraiment un LLM local en production

On ne va pas se mentir : les annonces marketing des éditeurs cloud vous font croire que l’IA locale, c’est soit un jouet pour geeks sous-équipés, soit une usine à gaz réservée aux GAFAM. La réalité ? Entre les deux. Et surtout, ça se mesure.

D’après les benchmarks récents de *Towards Data Science*, on peut enfin y voir clair : **un LLM local, ça coûte entre 0,10€ et 10€ par million de tokens**, selon l’architecture, le matériel et l’optimisation. Franchement, c’est moins que ce que beaucoup imaginent. Mais attention, le diable se cache dans les détails.

---

## Les fondements techniques : ce qui fait varier la note

Un LLM en local, c’est comme un moteur de Formule 1 dans une Clio : ça peut tenir la route, mais il faut adapter la carrosserie.

### L’architecture du modèle
Plus le modèle est gros, plus il bouffe de RAM, de VRAM et de CPU. Un **Mistral 7B** en 4 bits tient sur une RTX 4090 avec 24 Go de VRAM. Un **Llama 70B** ? Bonne chance avec ça sur une seule carte, même en 8 bits. Il faudra soit du multi-GPU, soit du CPU pur (et là, préparez le portefeuille).

Les **quantifications** changent la donne :
- **FP16** : précision maximale, mais gourmand en ressources.
- **INT8** : bon compromis, perte de qualité minime.
- **INT4** : on commence à sentir la compression, mais ça passe pour beaucoup d’usages.
- **INT3 ou moins** : là, on joue avec le feu. Les benchmarks montrent des chutes de performance significatives sur les tâches complexes.

Et puis il y a les **optimisations logicielles** : GGML, GGUF, EXL2, vLLM… Chaque format a ses forces. GGUF est devenu le standard pour les modèles quantifiés, mais vLLM excelle pour le serving multi-utilisateurs.

### Le matériel : le vrai nerf de la guerre
Une **RTX 4090** (24 Go VRAM) peut servir un Mistral 7B en INT4 sans sourciller. Mais pour du **Llama 13B**, il faudra soit :
- Une **A100 80 Go** (comptez 10 000€+ pour une machine complète)
- Ou un **cluster CPU** avec des Xeon ou des EPYC (et là, les coûts énergétiques explosent)

Les **accélérateurs dédiés** comme les TPU de Google ou les NPU (Neural Processing Units) des nouveaux PC commencent à changer la donne, mais ils restent marginaux pour le grand public.

## Implémentation : comment on passe de la théorie à la prod

### Déploiement solo vs. serving multi-utilisateurs
Un LLM en local pour un seul utilisateur, c’est simple : un script Python avec `transformers` ou `llama.cpp`, et hop. Mais dès qu’on veut servir plusieurs requêtes en parallèle, il faut :
- Un **serveur d’inference** (vLLM, TensorRT-LLM, ou même FastAPI + `transformers`)
- Un **système de batching** pour regrouper les requêtes
- Un **load balancer** si on scale horizontalement

Exemple concret avec **vLLM** :
```python
from vllm import LLM, SamplingParams

model = LLM(model="mistral-7b-instruct-v0.1", tensor_parallel_size=1)
sampling_params = SamplingParams(temperature=0.7, top_p=0.9)

outputs = model.generate(["Pourquoi le ciel est bleu ?"], sampling_params)
```

Simple, non ? Sauf que derrière, vLLM gère le **PagedAttention** pour optimiser la mémoire. Sans ça, un modèle de 7B peut saturer 14 Go de VRAM par utilisateur. Avec, on descend à 2-3 Go par requête.

### Les coûts cachés
- **Électricité** : Une RTX 4090 consomme 450W à pleine charge. À 0,20€/kWh, ça fait **0,09€/heure**. Sur un mois en continu, ça monte à 65€. Multipliez par le nombre de GPU…
- **Refroidissement** : Un rack de serveurs, ça chauffe. Comptez 10-20% de surcoût énergétique pour la clim.
- **Maintenance** : Les GPU meurent. Les disques SSD s’usent. Les mises à jour de drivers cassent tout. Prévoir un budget "merde, ça marche plus".

## Limitations : ce qui va vous faire rager

### La latence
Un LLM local, c’est lent. Même avec une 4090, un Mistral 7B met **0,5 à 1 seconde par token** en génération. Comparez ça aux 10-50 ms d’un API cloud comme Mistral AI ou OpenAI.

Pourquoi ? Parce que :
- Le **prétraitement** (tokenization) prend du temps.
- L’**attention** est séquentielle (même avec FlashAttention).
- Le **décodage** est token par token (sauf avec speculative decoding, mais c’est encore expérimental).

### La mémoire
Un modèle de 7B en FP16, ça fait **14 Go de VRAM**. En INT4, on descend à 3,5 Go. Mais :
- Le **contexte** (la fenêtre de tokens) mange de la RAM. 32k tokens en INT4 sur Mistral 7B, ça fait **~8 Go de VRAM supplémentaires**.
- Le **batch size** : si vous voulez traiter 10 requêtes en parallèle, multipliez la VRAM nécessaire par 10 (ou presque).

### La stabilité
Les modèles open source sont **moins optimisés** que ceux des GAFAM. Résultat :
- Des **crashs mémoire** si on pousse trop loin.
- Des **réponses moins bonnes** sur les edge cases (merci le fine-tuning maison).
- Des **problèmes de compatibilité** entre les versions de `transformers`, `torch`, et les kernels CUDA.

## Recherche et évolutions futures : ce qui va tout changer

### Les optimisations logicielles
- **FlashAttention 2/3** : déjà intégré dans `transformers`, réduit la latence de 20-30%.
- **Speculative Decoding** : permet de générer plusieurs tokens en parallèle, avec un gain de 2-3x en vitesse.
- **Multi-Query Attention (MQA)** : utilisé dans Llama 2, réduit la mémoire de 25% sans perte de qualité.

### Le matériel
- Les **GPU nouvelle génération** (Blackwell de NVIDIA, RDNA 4 d’AMD) promettent des gains de 50% en efficacité énergétique.
- Les **NPU** (comme ceux d’Intel ou Qualcomm) commencent à supporter l’inference LLM. Un jour, votre PC portable pourra faire tourner un 7B sans GPU.
- Les **accélérateurs open source** (comme ceux de [Cerebras](/articles/cerebras-en-europe-ce-que-valent-vraiment-ses-milliards-contre-nvidia--amateur)) pourraient casser les prix.

### Les modèles
- Les **modèles MoE (Mixture of Experts)** comme Mixtral ou Switch Transformers permettent de scaler sans exploser les coûts. Un Mixtral 8x7B a la puissance d’un 47B, mais avec l’empreinte mémoire d’un 13B.
- Les **modèles distillés** (comme TinyLlama ou Phi-3) offrent 80% des performances d’un gros modèle pour 20% du coût.
- Les **modèles on-device** (comme Gemma 2B de Google) sont optimisés pour tourner sur CPU avec des latences acceptables.

## FAQ

**[Un LLM local peut-il rivaliser avec les API cloud en termes de coût ?]**
Oui, mais seulement à partir d’un certain volume. En dessous de 10M tokens/mois, le cloud reste souvent plus économique. Au-delà, le local devient intéressant, surtout si on optimise bien le matériel et les modèles.

**[Quel est le modèle le plus rentable pour un déploiement local aujourd’hui ?]**
Mistral 7B en INT4 ou Llama 3 8B en INT4 sur une RTX 4090 offrent le meilleur compromis coût/performance. Pour du multi-utilisateurs, un Mixtral 8x7B sur 2x A100 est un bon choix.

**[Quelles sont les principales erreurs à éviter en déployant un LLM local ?]**
Sous-estimer la VRAM nécessaire (surtout avec le contexte), négliger le refroidissement, et oublier que les modèles open source nécessitent plus de tuning que les API clés en main. Et surtout, ne pas benchmarker avant d’acheter du matériel.
